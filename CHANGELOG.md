### Added

#### `wiki/relying_party_requirements.md` *(new file)*

Comprehensive technical reference for Relying Party (Verifier) implementers.  Covers all three
presentation flows supported by this wallet:

- **Section 1 – JOSE and X5C background:** Explains the JOSE standard family (JWS, JWE, JWK, JWT,
  JWA), the `x5c` JOSE header parameter (RFC 7515 §4.1.6), and the JAR (JWT-Secured Authorization
  Request, RFC 9101) pattern.  Includes links to all relevant RFCs.
- **Section 2 – Supported Client Identifier Schemes:** Documents the two schemes enabled in
  `WalletCoreConfigImpl` — `x509_san_dns` (`ClientIdScheme.X509SanDns`) and `x509_san_hash`
  (`ClientIdScheme.X509Hash`) — with the exact Kotlin configuration.
- **Section 3 – x509_san_dns vs x509_san_hash:** Side-by-side comparison table covering trust
  mechanism, `client_id` content, certificate rotation impact, hostname verification requirements,
  and suitability for production vs. local development.
- **Section 4 – X.509 Certificate Profile:** ISO 18013-5 `ProfileValidation` requirements for the
  leaf certificate including mandatory extensions (SKI, AKI, `mdlReaderAuth` OID `1.0.18013.5.1.6`,
  `digitalSignature` Key Usage), prohibited critical extensions, signature algorithm constraints
  (ECDSA only), and the 1 187-day validity limit.
- **Section 5 – Trust Anchor Configuration:** Documents the nine bundled IACA PEM files, the shared
  `ReaderTrustStore` (proximity + remote OpenID4VP), and the distinct TLS trust path (Android system
  CA store).
- **Section 6 – Same-Device Flow:** Mermaid sequence diagram, URI scheme table (`haip-vp://`,
  `openid4vp://`, `eudi-openid4vp://`, `mdoc-openid4vp://`), full Authorization Request parameter
  table, Request Object signing requirements, TLS requirements, DocType/VCT tables for mDOC and
  SD-JWT VC, response endpoint requirements, redirect URI handling, and verifier trust indication.
- **Section 7 – Cross-Device OpenID4VP Flow:** Mermaid sequence diagram, QR code requirements,
  `request_uri` endpoint requirements, response endpoint differences vs. same-device.
- **Section 8 – Cross-Device Proximity Flow (ISO 18013-5 BLE/NFC):** Mermaid sequence diagram,
  Device Engagement QR code explanation (generated by the wallet, not the RP), Reader Authentication
  requirements, IACA trust store PEM file table, BLE/NFC transport requirements, and the mdoc
  Request CBOR structure.
- **Section 9 – Credential Format Requirements:** mDOC and SD-JWT VC format table; one-time-use
  credential policy note.
- **Section 10 – Credential Revocation Behaviour:** Periodic revocation check interval; valid /
  revoked / suspended document handling.
- **Section 11 – Summary of Hard Requirements:** Table of 19 hard requirements (R1–R19) with flow
  scope.
- **Section 12 – Common Error Messages:** Lookup table of `InvalidJarJwt`, `CertPathValidatorException`,
  `vp_formats` missing, algorithm mismatch, TLS failure — with causes and fixes.
- **Section 13 – Known Limitations and GitHub Issues:** References to upstream issues #500, #230,
  #247, #439.
- **Section 14 – Glossary:** 35+ terms (HAIP, JOSE, JWS, JWT, JAR, x5c, client_id, IACA,
  ReaderAuthentication, DeviceEngagement, BLE, HCE, PID, etc.).
- **Section 15 – Reference Specifications:** Links to OpenID4VP 1.0, HAIP profile, ISO 18013-5/7,
  all relevant RFCs, DIF PE 2.0, SD-JWT VC, EUDI ARF.

#### `wiki/local_network.md` *(new file)*

Step-by-step guide for running a local credential verifier + the EUDI wallet on a private network
with self-issued certificates and no dependency on public infrastructure.

- **Mermaid architecture diagram** (topology: Mac verifier ↔ Wi-Fi router ↔ Android device) and
  **Mermaid sequence diagram** of the full local end-to-end flow.
- **`x509_san_hash` analysis:** Explains why `x509_san_hash` is the correct scheme for local
  development, and documents the wallet-core ≤ 0.25 caveat (shared `ReaderTrustStore` still
  enforces `ProfileValidation` even for remote OpenID4VP flows).
- **`generate_local_certs.sh` bash script:** Self-contained `openssl` script that generates a
  Root CA and a leaf certificate satisfying all ISO 18013-5 `ProfileValidation` requirements
  (`mdlReaderAuth` OID, SKI/AKI, ECDSA P-256, ≤ 1 187 days validity, TLS SANs for local IP).
  Auto-detects the Mac's Wi-Fi IP.  Prints the ready-to-paste `x509_san_hash:` `client_id` value.
- **Credential verifier TOML configuration** example (`credential-server.toml`) with local cert
  paths and `client_id` placeholder.
- **Wallet configuration steps:** How to add the Root CA to `ReaderTrustStore`, how to replace
  the production `HttpClient` with a dev-only trust-all variant for TLS (with security warning).
- **Android emulator note** (`10.0.2.2` address).
- **End-to-end test sequence diagram.**
- **Troubleshooting section** with targeted `openssl` commands for each failure mode.

### Changed

#### `common-feature/src/main/java/eu/europa/ec/commonfeature/ui/qr_scan/QrScanViewModel.kt`

Added structured debug logging of the scanned QR URL to aid local development and debugging.

- **New dependency:** `LogController` injected into `QrScanViewModel` via Koin (already available
  to all `project.android.feature` modules through `AndroidFeatureConventionPlugin`; no
  `build.gradle.kts` change required).
- **New constant:** `EUDI_QR_LOG_TAG = "EUDI_QR_SCAN"` — a stable log tag filterable in Logcat.
- **New log statement** in `handleScannedQr()`, executed immediately after the URL passes
  `validateForm()` and before `calculateNextStep()`:
  ```kotlin
  logController.d(EUDI_QR_LOG_TAG) { "QR scanned – URL: $scannedQr" }
  ```
  To filter in Logcat:
  ```
  adb logcat -s EUDI_QR_SCAN
  ```
  or use the tag `EUDI_QR_SCAN` in the Android Studio Logcat filter.
